2. CI/CD PIPELINE (GitHub Actions)
в”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓ

Every push to a feature branch triggers:
  вњ“ Lint (ruff, mypy)
  вњ“ Unit tests (pytest)
  вњ“ Security scan (bandit, pip-audit)
  вњ“ Docker build (verifies image builds successfully)

Merge to `main` additionally triggers:
  вњ“ Integration tests
  вњ“ Auto-deploy to staging
  вњ“ Slack notification to #deployments

Release to production:
  1. Create a GitHub release with a semantic version tag: v2.x.y
  2. The "Deploy to Prod" workflow triggers automatically
  3. Requires approval from: Engineering Manager + one senior engineer
  4. Cloud Build builds and pushes the image to Artifact Registry
  5. Cloud Run deploys the new revision with 0% traffic
  6. Automated smoke tests run against the new revision
  7. If tests pass, traffic is shifted 10% в†’ 50% в†’ 100% over 15 minutes
  8. If any smoke test fails, traffic shifts back to previous revision

3. DEPLOYING MANUALLY (EMERGENCY / HOTFIX)
в”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓ

# Build and push the image
PROJECT=hermes-agent-prod
REGION=us-central1
IMAGE=gcr.io/$PROJECT/hermes-gateway:hotfix-$(date +%Y%m%d%H%M)

docker build -f Dockerfile.gateway -t $IMAGE .
docker push $IMAGE

# Deploy to Cloud Run (creates new revision with 0% traffic first)
gcloud run deploy hermes-gateway \
  --image=$IMAGE \
  --region=$REGION \
  --project=$PROJECT \
  --no-traffic

# Verify the new revision
NEW_REV=$(gcloud run revisions list --service=hermes-gateway \
  --region=$REGION --project=$PROJECT \
  --sort-by="~creationTimestamp" --limit=1 --format="value(name)")
curl https://$NEW_REV---hermes-gateway-uc.a.run.app/health

# Shift traffic (if healthy)
gcloud run services update-traffic hermes-gateway \
  --region=$REGION --project=$PROJECT \
  --to-revisions=$NEW_REV=100

Feature flags are managed via GCP Feature Flags (Cloud Deploy) and read
from Secret Manager at startup. To toggle a flag:
  1. Update the flag value in Secret Manager: sm://hermes-agent-prod/feature-flags
  2. Restart Cloud Run to pick up the new value (or use the /admin/reload endpoint)

  ENABLE_MULTI_PROVIDER_LLM=true # LiteLLM provider selection

Prerequisites: Python 3.11+, Docker, gcloud CLI, Node.js 20+

  # Clone and set up
  git clone git@github.com:acmecorp/hermes-gcp.git
  cd hermes-gcp
  python -m venv .venv && source .venv/bin/activate
  pip install -r requirements.txt
  cp .env.example .env  # Fill in your GCP values

  # Authenticate
  gcloud auth application-default login
  gcloud config set project hermes-agent-prod

  # Run the gateway locally
  uvicorn gateway.main:app --reload --port 8080

  # Run the Next.js UI
  cd ui && npm install && npm run dev

  # Run tests
  pytest tests/ -v

7. API STANDARDS
в”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓв”Ѓ
  - All endpoints return JSON.
  - HTTP status codes: 200 OK, 202 Accepted (async), 400 Bad Request,
    401 Unauthorized, 403 Forbidden, 404 Not Found, 429 Too Many Requests,
    500 Internal Server Error.
  - Pagination: use cursor-based pagination for list endpoints (next_token).
  - Rate limits: 20 req/min for chat, 5 req/min for tasks (per IP).
  - Auth: all endpoints require Authorization: Bearer <Google ID token>
    except /health (public) and /scheduler/trigger (OIDC token).
